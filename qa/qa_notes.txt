I was investigating why p2p-fullblocktest.py was taking so long to complete. The test continues until block 88 fairly quickly, and then gets to this part of the test:

```
        #  Test re-org of a week's worth of blocks (1088 blocks)
        #  This test takes a minute or two and can be accomplished in memory
        #    
        if self.options.runbarelyexpensive:
            tip(88)
```

the python mininode then builds 1088 blocks. Let's call them B1 - B1088. It's smart enough to not send 1088 individual inv messages with a single block header hash, but instead sends a single inv message with all 1088 header hashes. The bitcoin node then sends 1088 individual GetHeaders messages, one for each header hash (since the node doesn't know anything about how those different headers hashes are in a chain, just that there are 1088 blocks that it doesn't already have). Let's call those messages GetHeader(B1), GetHeader(B2), ...

The python mininode responds to the GetHeaders as follows:

- To GetHeader(B1), it responds with a Headers message containing B1. Let's call this Headers(B1)
- To GetHeader(B2), it responds with a Headers message containing B2 and B1. Let's call this Headers(B2, B1)
- To GetHeaders(B3), it responds with Headers(B3,B2,B1)
...
- To GetHeaders(B1088), it responds with Headers(B1088,B1087,...,B1)

ie the header for B1 is sent 1088 times, the header for B2 is sent 1087 times, and so on until the header for B1088 is sent once. The total number of headers sent is therefore 1088 * (1088 - 1) ~= 1,200,000. At 81 bytes each,
